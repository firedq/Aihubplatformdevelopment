# Epic 6：智能通知与消息触发

**目标**：表单管理员能够配置基于表单字段状态变化的自动通知规则，系统在触发条件满足时自动向供应商联系人和内部人员发送短信（P0）或邮件（P1）通知。

**FRs covered:** FR30, FR31, FR32, FR33, FR34

**相关 NFR：** NFR5（通知发送延迟 ≤10 分钟）、NFR7（通知服务凭证加密存储）、NFR23（短信服务商可配置切换）

**Stories:**
- Story 6.1: 配置字段状态变化触发规则
- Story 6.2: 配置通知发送对象与渠道
- Story 6.3: 自定义通知文案模板与动态变量
- Story 6.4: 自动触发通知发送

---

## 业务规则

| # | 规则名称 | 条件 | 结果 |
|---|---------|------|------|
| BR-6.1 | 字段类型与条件匹配 | 文本型字段 | 条件选项仅显示"等于/不等于/包含"；数值型字段显示全部条件（含大于/小于） |
| BR-6.2 | 规则状态控制 | 规则被禁用时 | 该规则不参与条件评估，不触发通知；其他启用规则不受影响 |
| BR-6.3 | 通知幂等发送 | 同一 submissionId + ruleId 重复触发 | 通过幂等校验，只发送一次通知，不产生重复短信 |
| BR-6.4 | 渠道前置依赖 | 选择短信渠道时 | 应用须已完成短信配置（Story 3.4）；未配置时渠道选项置灰 |
| BR-6.5 | 通知失败不阻塞 | 短信发送失败 | 按指数退避重试（30s → 2min），最多 2 次；仍失败记录日志，不进入死信队列，不影响 OA 数据回传主流程 |
| BR-6.6 | 变量空值处理 | 通知模板中变量对应字段值为空 | 占位符替换为空字符串，最终文案语义尽量完整 |
| BR-6.7 | 短信长度分片 | 短信内容超过 70 字 | 自动拆分为多条短信发送，每条费用独立计费 |
| BR-6.8 | 通知延迟上限 | 从触发条件满足到通知发送 | 整体延迟 ≤10 分钟（NFR5）|

## 待确认事项

1. **触发时机**：通知规则的触发时机是"供应商提交时立即评估"，还是"数据推送至 OA 后评估"？如果数据未推送至 OA（因为未到推送时间），通知是否仍应发送？
2. **多条规则触发顺序**：同一供应商提交时，若多条规则同时满足触发条件，通知发送顺序是否有优先级规则？是并发发送还是串行？
3. **发送对象的动态手机号**：选择"指定供应商联系人"时，手机号从哪里取？是表单中的手机号字段，还是供应商 SSO 身份中的手机号？
4. **邮件模板富文本支持**：P1 阶段邮件模板是否支持 HTML 富文本（图片/表格/格式化），还是仅纯文本 + 动态变量？
5. **通知记录查看权限**：通知日志是否对"表单管理员"可见，还是只有"数据管理员"才能查看（影响 Epic 7 Story 7.x 的设计）？

### Story 6.1：配置字段状态变化触发规则

As a 表单管理员，
I want 为表单配置基于字段值变化的通知触发规则（如"报价金额 > 100000 时触发"），
So that 系统能够在关键业务条件满足时自动发出通知，无需人工监控。

**前提条件：**
- 表单管理员已进入应用工作台（Story 2.3）
- 表单已发布（Story 4.6）
- 表单中已有可选择的字段

**操作步骤：**
1. 在应用工作台的"通知规则"模块，点击"新增触发规则"
2. 选择触发字段（从当前表单字段下拉列表选择）
3. 选择触发条件（根据字段类型动态显示：等于/不等于/包含/大于/小于/大于等于/小于等于）
4. 输入触发值
5. 点击"保存规则"

**成功标志：**
- 规则保存成功，规则列表展示：触发字段、条件表达式、创建时间、启用状态（默认启用）
- 数值型字段显示全部条件选项，文本型字段仅显示等于/不等于/包含
- 可对规则进行启用/禁用切换

**页面描述：**
- **通知规则配置页**：规则列表（含已有规则的卡片，每张卡片显示：触发字段名、条件表达式、渠道标签、状态开关、编辑/删除按钮）+ 顶部"新增触发规则"按钮
- **新增规则抽屉/弹窗**：
  - 第一步：选择触发字段（下拉选择，来自当前表单字段列表）
  - 第二步：选择条件（根据字段类型动态显示可选条件）+ 输入触发值
  - 第三步：配置发送对象（见 Story 6.2）+ 选择渠道
  - 第四步：编辑通知文案（见 Story 6.3）
  - 保存按钮

**Acceptance Criteria:**

**Given** 表单管理员在已发布表单的"通知规则"配置页
**When** 点击"新增触发规则"
**Then** 展示规则配置表单：选择触发字段（从当前表单字段列表选择）、选择条件（等于 / 不等于 / 包含 / 大于 / 小于 / 大于等于 / 小于等于）、输入触发值；支持同时配置多个规则（满足 FR30）

**Given** 表单管理员配置触发规则"字段'报价金额' > 100000"
**When** 点击保存
**Then** 规则保存成功，规则列表展示：触发字段、条件表达式、创建时间、启用状态（默认启用）

**Given** 表单管理员配置触发规则
**When** 选择的字段类型为非数值型（如文本字段）
**Then** 条件选项中的"大于 / 小于"选项置灰不可选，仅显示"等于 / 不等于 / 包含"

**Given** 已创建的触发规则
**When** 管理员切换规则为"禁用"
**Then** 该规则立即停止触发通知，状态更新为"已禁用"，不影响其他启用中的规则

**异常情况：**
- 如果触发值为空（规则配置不完整），提示"请填写触发值"，阻止保存
- 如果同一字段已存在完全相同条件的规则（字段+条件+触发值均相同），提示"已存在相同规则，是否继续创建？"（警告，不阻止）

### Story 6.2：配置通知发送对象与渠道

As a 表单管理员，
I want 为通知规则配置发送对象（供应商联系人手机号或内部人员角色）和发送渠道（短信 / 邮件），
So that 通知能够精准到达需要知晓的相关方。

**前提条件：**
- 触发规则已创建（Story 6.1）或正在创建规则的第二步
- 如需短信渠道，应用已完成短信配置（Story 3.4）
- 如需邮件渠道（P1），应用已完成邮件配置（Story 3.5）

**操作步骤：**
1. 在规则配置的"发送对象"区域，选择发送对象类型
   - 选项 A：指定供应商联系人（输入手机号列表，或从供应商链接列表选择）
   - 选项 B：指定内部人员角色（从 RBAC 角色列表选择）
2. 在"发送渠道"区域，选择短信（P0）或邮件（P1）或两者（双渠道）
3. 确认后进入通知文案配置（Story 6.3）

**成功标志：**
- 发送对象配置保存成功，预览显示发送目标数量
- 渠道选择后关联已配置的对应服务，可进行文案配置

**页面描述：**
- **发送对象区域**：类型选择 Radio（指定手机号/指定角色）→ 动态显示对应输入区
  - 指定手机号：多行文本输入（逗号分隔）或从供应商列表多选（Checkbox 列表）；提示"最多 20 个"
  - 指定角色：角色下拉单选（数据来自 RBAC 角色列表）；说明"将发送给该应用内所有拥有此角色的用户"
- **发送渠道区域**：Checkbox（可多选）：短信（依赖 Story 3.4 配置）/ 邮件-P1（依赖 Story 3.5 配置）；未配置的渠道显示置灰 + "去配置"链接

**Acceptance Criteria:**

**Given** 表单管理员在通知规则的"发送对象"配置区
**When** 选择发送对象类型
**Then** 提供两种选项：
  1. **指定供应商联系人**：输入手机号列表（逗号分隔，最多 20 个）或从供应商链接列表中选择供应商（自动取其绑定手机号）
  2. **指定内部人员角色**：从 RBAC 角色列表选择（如"应用配置管理员"），通知发送给该应用下拥有该角色的所有用户（满足 FR31）

**Given** 表单管理员选择发送渠道
**When** 应用已完成短信配置（Story 3.4 通过）
**Then** 短信渠道可选（P0），选择后关联已配置的短信服务（满足 FR32 P0）

**Given** 表单管理员选择发送渠道
**When** 应用已完成邮件配置（Story 3.5 通过，P1）
**Then** 邮件渠道可选（P1），选择后关联已配置的邮件服务（满足 FR32 P1）

**Given** 通知渠道（短信或邮件）未在当前应用中配置
**When** 管理员尝试选择该渠道
**Then** 渠道选项置灰，并显示提示"请先完成{短信/邮件}服务配置"，附跳转链接至 Story 3.4 / 3.5 配置页

**Given** 管理员同时选择短信和邮件渠道（P1 应用双渠道）
**When** 触发条件满足时
**Then** 系统分别向两个渠道发送通知，各自记录发送状态

**异常情况：**
- 如果指定的供应商手机号列表全部无效（格式错误），保存时提示"至少需要一个有效手机号"，阻止保存
- 如果选择的角色在当前应用下无用户，提示"该角色下当前无用户，通知将无法发送"（警告，允许保存）

### Story 6.3：自定义通知文案模板与动态变量

As a 表单管理员，
I want 自定义通知的文案内容，并在模板中插入表单动态变量（如供应商名称、报价金额），
So that 收到通知的人员能够直接从通知内容中了解关键信息，无需再次登录查询。

**前提条件：**
- 规则触发条件和发送对象已配置（Story 6.1 + 6.2）
- 应用已完成短信配置（Story 3.4），如需邮件也已完成邮件配置（Story 3.5）

**操作步骤：**
1. 在规则配置的"通知文案"区域，选择编辑短信/邮件模板
2. 在文本编辑框中输入通知正文
3. 点击"插入变量"按钮，从表单字段列表选择需要插入的变量（如"报价金额"、"供应商名称"）
4. 选中变量后自动插入占位符（如 `{{报价金额}}`）到光标位置
5. 点击"预览"，查看用示例数据替换变量后的实际效果
6. 确认文案后保存规则

**成功标志：**
- 变量占位符正确插入文案
- 预览时示例数据正确替换占位符
- 短信超过 70 字时显示分片提示

**页面描述：**
- **通知文案配置区**：多行文本编辑框（短信模板）；右上角字符计数（当前字数/70，超出显示橙色警告）；工具栏含"插入变量"按钮 → 点击弹出表单字段下拉列表（含系统变量：{{提交时间}}、{{供应商名称}}等）；"预览"按钮 → 弹出预览弹窗，用示例值替换占位符展示实际文案

**系统内置变量（所有表单通用）：**
| 变量名 | 说明 |
|--------|------|
| `{{提交时间}}` | 供应商提交时间（精确到分钟）|
| `{{表单名称}}` | 当前表单名称 |
| `{{供应商名称}}` | SSO 身份中的供应商名称 |
| `{{截止时间}}` | 表单填写截止时间 |

**Acceptance Criteria:**

**Given** 表单管理员在通知规则的"通知内容"配置区
**When** 编辑短信模板文案
**Then** 文本编辑框支持插入动态变量：点击"插入变量"按钮，弹出当前表单字段列表，选择后自动插入变量占位符（如 `{{报价金额}}`、`{{供应商名称}}`、`{{提交时间}}`）（满足 FR33）

**Given** 管理员完成模板编辑（如"尊敬的采购员，供应商{{供应商名称}}已提交报价{{报价金额}}元，请登录系统查看详情。"）
**When** 点击"预览"
**Then** 系统用示例数据替换变量占位符，展示实际发送内容预览

**Given** 模板中包含变量占位符
**When** 触发通知发送时该字段值为空
**Then** 对应占位符替换为空字符串（不显示"null"或"undefined"），最终文案仍语义完整

**Given** 短信模板字符数超过运营商限制（单条短信 70 字）
**When** 模板编辑器检测到字符数超限
**Then** 显示字数提示（如"已超出 70 字，将拆分为 2 条短信发送"），不阻止保存，但发送时自动分片

**Given** 邮件模板配置（P1）
**When** 管理员编辑邮件模板
**Then** 支持配置邮件主题（含动态变量）和邮件正文（富文本，含动态变量），预览效果与实际邮件内容一致

**异常情况：**
- 如果通知文案为空直接保存，提示"通知文案不能为空"，阻止保存
- 如果插入了不存在的变量占位符（手动输入，非系统提供），发送时保留原始占位符文字（不做替换），并在发送前告警提示管理员

### Story 6.4：自动触发通知发送

As a 系统，
I want 在通知触发条件满足时自动向指定对象发送通知，
So that 无需人工干预，相关方能够及时获知关键业务状态变化。

**前提条件：**
- 通知规则已完整配置（Story 6.1 + 6.2 + 6.3）
- 短信服务已配置（Story 3.4）
- RabbitMQ `aihub.notify` 队列已就绪

**系统执行流程：**
1. 供应商提交表单 → 数据写入 AI-Hub 数据库
2. 系统逐条评估所有启用中的通知规则
3. 条件匹配的规则：通知消息入队 `aihub.notify`
4. 通知消费者从队列取出消息，调用 `SmsClientFactory` 发送短信
5. 发送成功：记录成功日志（含 MessageId）
6. 发送失败：指数退避重试（30s → 2min），最多 2 次；仍失败记录失败日志

**成功标志：**
- 触发条件满足后 ≤10 分钟内，目标手机号收到短信通知
- `notification_log` 记录每次通知的触发时间、发送状态、MessageId
- 重复提交场景下，同一规则仅发送一次通知

**Acceptance Criteria:**

**Given** 供应商提交表单数据后
**When** 系统评估所有启用中的通知规则（逐条条件匹配）
**Then** 条件匹配的规则触发通知任务，将通知消息入队至 `aihub.notify` RabbitMQ 队列（满足 FR34）

**Given** 通知消息在 `aihub.notify` 队列中
**When** 通知消费者取出消息并发送短信
**Then** 通过 `SmsClientFactory` 调用已配置短信服务商，发送成功后：`notification_log` 表记录状态=成功、发送时间、MessageId；整体通知延迟从触发到发送 ≤10 分钟（NFR5）

**Given** 短信发送失败（余额不足 / 服务商超时 / 手机号格式错误）
**When** 消费者捕获发送异常
**Then** 按指数退避重试最多 2 次（30s → 2min），仍失败后记录状态=失败、错误码、错误原因至 `notification_log`；不进入死信队列（通知失败不影响核心数据回传流程）

**Given** 同一供应商提交因网络重传触发多次规则评估
**When** 同一提交 submissionId 重复触发相同规则
**Then** 通过 `submissionId + ruleId` 幂等校验，重复触发只发送一次通知，不产生重复短信

**Given** 数据管理员查看通知记录
**When** 进入通知日志页面（见 Epic 7 Story 7.x）
**Then** 可查看每条通知的触发时间、触发规则、发送对象、发送状态、失败原因（若失败）

**异常情况：**
- 如果 `aihub.notify` 队列服务临时不可用，通知消息暂存至本地内存队列，RabbitMQ 恢复后补发，最大容忍延迟 30 分钟
- 如果规则评估发生异常（如字段类型转换错误），记录异常日志并跳过该规则，不影响其他规则评估
- 如果通知发送目标手机号在短信服务商黑名单中，发送失败，记录"手机号被拦截"原因，不重试
