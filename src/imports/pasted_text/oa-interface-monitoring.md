## 业务规则

| # | 规则名称 | 条件 | 结果 |
|---|---------|------|------|
| BR-7.1 | 失败记录重试方式 | 数据管理员手动触发重试 | 失败记录重新入队（重置重试计数，不走 DLQ 通道），无手动重试次数上限 |
| BR-7.2 | 批量重试上限 | 单次批量重试操作 | 最多同时入队 500 条（NFR18），超出时提示分批执行 |
| BR-7.3 | 审计日志追加写 | 审计日志表设计原则 | 仅支持 INSERT + SELECT，无 UPDATE/DELETE 接口；数据库层面限制操作权限 |

| BR-7.8 | 提交状态定义 | 填写数据明细中供应商状态 | 未访问（链接已发送但未打开）/ 已访问未提交 / 已提交 / 已修改 |

### Story 7.2：接口失败数据一键重试

As a 数据管理员，
I want 对失败的 OA 回传数据执行一键重试，
So that 因临时性故障（OA 短暂宕机、网络抖动）导致的失败记录无需人工介入即可恢复回传。

**前提条件：**
- 数据管理员已进入接口监控页面（Story 7.1）
- 存在状态为"最终失败"的记录（进入 DLQ）

**操作步骤：**
1. 在失败明细列表中，勾选需要重试的记录（单条或多条）
2. 点击"重试"按钮（单条）或"批量重试"按钮（多条）
3. 系统确认提示（"确认重试 N 条记录？"）
4. 确认后，选中记录重新入队，状态变为"重试中"
5. 等待消费者处理完成，刷新页面查看最新状态

**成功标志：**
- 记录成功重新入队，状态变为"重试中"
- 重试成功后，状态变为"成功"，统计概览数字刷新（失败量减少）
- 重试再次失败，状态回到"最终失败"，失败次数+1

**页面描述：**
- **失败明细列表操作区**：列表左侧 Checkbox 列（全选/单选）；顶部浮出"已选 N 条"提示 + "批量重试"按钮；单条记录操作列有"重试"按钮
- **重试确认弹窗**："确认重试 N 条记录？重试将重新推送数据至 OA。" + 确认/取消按钮

**Acceptance Criteria:**

**Given** 数据管理员在失败记录列表中选中一条或多条记录
**When** 点击"重试"按钮
**Then** 选中记录重新入队至 `aihub.oa.write`（不走 DLQ，重置重试计数），记录状态更新为"重试中"，页面刷新后显示最新状态（满足 FR37）

**Given** 数据管理员点击"全部重试"
**When** 当前筛选条件下存在多条失败记录
**Then** 所有符合条件的失败记录批量入队，同时显示"共提交 N 条重试任务"提示；批量重试单次上限 500 条（NFR18）

**Given** 重试成功（OA 返回成功响应）
**When** 消费者处理完成
**Then** `interface_call_log` 记录状态更新为"成功"，失败明细列表中该记录消失（或状态变更为"已成功"），统计概览数字同步刷新

**Given** 重试再次失败
**When** 重试任务执行后 OA 仍返回错误
**Then** 记录状态回到"最终失败"，重试次数+1，失败原因更新为最新错误信息；数据管理员可再次手动重试，无次数上限（人工操作不受自动重试 3 次限制）

**异常情况：**
- 如果选择了超过 500 条记录批量重试，提示"单次批量重试上限 500 条，请筛选后分批执行"，阻止操作
- 如果重试触发后 OA 接口仍处于故障中（所有重试均失败），数据管理员可查看最新错误信息，决定等待 OA 恢复后再试

### Story 7.5：关键操作审计日志

As a 系统，
I want 记录关键操作的审计日志（表单发布 / 接口配置变更 / 数据回传 / AI 积分消耗），
So that 所有关键操作有据可查，满足合规审计要求，且日志不可篡改，保留 ≥180 天。

**前提条件：**
- 审计日志功能在系统初始化时自动启用（无需手动配置）
- 审计日志表为追加写（INSERT ONLY），数据库权限已限制 UPDATE/DELETE

**系统执行规则（自动记录的操作类型）：**
| 操作类型 | 触发时机 | 记录内容示例 |
|---------|---------|-------------|
| 表单发布/下线 | 管理员点击发布/下线按钮 | 操作人、表单ID、操作前状态→后状态 |
| OA 配置变更 | 保存 OA 集成配置 | 操作人、变更字段名（不含凭证值）、变更前后值 |
| OA 数据回传 | 队列消费者处理每条消息 | 消息ID、供应商标识、OA返回码、耗时 |
| AI 指令执行 | AI 服务调用完成 | 操作人、指令内容摘要、Token消耗量 |
| AI 积分扣减 | 积分扣减事务完成 | 租户ID、用户ID、扣减量、余额 |
| 用户角色变更 | 超级管理员修改用户权限 | 操作人、被操作用户、变更前后角色 |

**操作步骤（数据管理员查询）：**
1. 在应用工作台，进入"审计日志"查询页
2. 按操作人/操作类型/时间范围/操作对象进行筛选
3. 查看符合条件的日志列表（分页显示）
4. 需要时，点击"导出 CSV"

**Acceptance Criteria:**

**Given** 任何用户执行以下操作之一
**When** 操作完成（成功或失败）
**Then** 系统自动记录审计日志，包含字段：操作人（userId + 用户名）、操作时间（精确到秒）、操作类型（枚举：表单发布 / 表单下线 / OA 配置变更 / OA 数据回传 / AI 指令执行 / AI 积分扣减 / 用户角色变更）、操作对象（appId / formId / userId）、操作前后数据 Snapshot（JSON diff）（满足 FR41）

**Given** 审计日志已写入
**When** 任何程序尝试通过 API 或数据库直接修改日志记录
**Then** 审计日志表设计为追加写（INSERT ONLY），无 UPDATE/DELETE 接口暴露；数据库权限仅允许 INSERT 和 SELECT（NFR11）

**Given** 系统审计日志
**When** 审计日志创建时间超过 180 天
**Then** 系统自动归档至冷存储（不删除），归档后仍可通过管理端查询，查询响应时间可接受（≤10 秒，NFR11 保留周期 ≥180 天）

**Given** 数据管理员在审计日志查询页
**When** 按操作人、操作类型、时间范围、操作对象进行筛选查询
**Then** 返回符合条件的日志列表，分页展示（每页 20 条），支持导出 CSV

**Given** AI 积分扣减操作
**When** 表单管理员执行 AI 指令（截图解析 / 自然语言编辑）
**Then** 审计日志记录本次操作消耗的 Token 量、AI 服务商返回的 usage 数据，与 Epic 8 积分系统联动

**异常情况：**
- 如果审计日志写入失败（数据库异常），记录异常至系统错误日志，并触发告警（不能因为审计日志写入失败而阻塞业务操作本身）
- 如果查询归档日志超过 10 秒，展示"数据加载中"进度提示，最长等待 30 秒后超时提示
