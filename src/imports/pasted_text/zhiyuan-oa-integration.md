# Epic 3：致远 OA 系统双向集成

**目标**：（在某个应用内）应用配置管理员能够配置与致远 OA 的双向 API 对接，验证连通性，使供应商提交的数据通过异步队列自动回传并触发 OA 审批流程，并配置短信/邮件通知接口。

**FRs covered:** FR8, FR9, FR10, FR11, FR12, FR13

**相关 NFR：** NFR4（OA 回传 ≤10min）、NFR7（OA 凭证 AES-256 加密）、NFR9（凭证不暴露/不写日志）、NFR14（回传成功率 ≥99%，最多重试 3 次）、NFR20（支持 A8 V5 8.0+）、NFR23（短信服务商可配置切换）

**Stories:**
- Story 3.1: 致远 OA 接口配置（读取/写入方向）
- Story 3.2: OA 接口连通性测试
- Story 3.3: 供应商数据异步回传 OA 队列
- Story 3.4: 短信通知服务接口配置
- Story 3.5: 邮件通知服务接口配置（P1）

## 业务规则

| # | 规则名称 | 条件 | 结果 |
|---|---------|------|------|
| BR-3.1 | 凭证加密存储 | OA API Key/Token 保存时 | AES-256 加密存入 DB，前端展示为掩码（`****8a3f`），接口响应不返回明文 |
| BR-3.2 | 凭证不写日志 | OA 凭证在任何场景下 | 系统日志中不记录 OA API Key/Token 明文，避免日志泄露 |
| BR-3.3 | 接口版本适配 | 致远 OA 版本为 A8 V5 8.0+（P0）或 A6（P1）| 系统根据配置的版本选项切换对应适配器（ZhiyuanA8V5Adapter / ZhiyuanA6Adapter） |
| BR-3.4 | 异步回传防阻塞 | 供应商提交表单后 | 数据立即入队 `aihub.oa.write`，不等待 OA 响应，供应商端立即收到"提交成功"确认 |

| BR-3.6 | 幂等防重复 | 同一条数据因网络重传被重复入队 | 通过 `messageId`（UUID）Redis 幂等校验，重复消息直接 ACK，不产生重复 OA 写入 |
| BR-3.7 | 短信服务商热切换 | 管理员切换短信服务商 | 通过 `SmsClientFactory` 动态切换，无需重启服务 |
| BR-3.8 | 回传延迟上限 | 正常队列下 OA 回传 | 整体回传延迟（含队列等待）≤10 分钟（NFR4） |

### Story 3.1：致远 OA 接口配置（读取/写入方向）

As a 应用配置管理员，
I want 在应用内配置与致远 OA 的 API 对接，指定关联的 OA 表单名称、表单 ID，并设定数据流动方向，
So that 平台知道从哪个 OA 表单读取字段结构，以及将供应商提交的数据写入哪个 OA 接口。

**前提条件：**
- 应用配置管理员已登录，且已进入某个应用的工作台（Story 2.3）
- 已获取致远 OA 的服务地址、API Key/Token 和目标表单 ID（由企业 IT 部门提供）

**操作步骤：**
1. 进入应用工作台的"OA 集成配置"模块
2. 填写致远 OA 服务地址（如 `http://oa.company.com`）
3. 填写 API Key 或 Token
4. 填写关联的 OA 表单名称及表单 ID
5. 选择数据流动方向（读取 OA / 写入 OA / 双向）
6. 选择 OA 版本（A8 V5 / A6）
7. 点击"保存配置"

**成功标志：**
- 配置保存成功，OA API Key/Token 加密存储
- 再次进入配置页，凭证显示为掩码（如 `****8a3f`）
- OA 版本选择正确对应的适配器

**页面描述：**
- **OA 集成配置页**：分区域显示：连接参数区（OA 服务地址、API Key/Token、版本选择）、表单绑定区（表单名称、表单 ID、数据流向选择 Radio）、操作按钮区（保存配置、测试连通性）
- **凭证字段**：输入时显示明文，保存后重新加载时显示掩码，旁边有"修改"按钮重新输入

**Acceptance Criteria:**

**Given** 应用配置管理员进入某应用的 OA 集成配置页
**When** 填写致远 OA 服务地址、API Key/Token、关联 OA 表单名称及表单 ID，选择数据流向（读取 OA / 写入 OA / 双向）
**Then** 配置保存成功，OA API Key/Token 以 AES-256 加密存入数据库，前端不展示明文，接口响应中不返回密钥值

**Given** OA 配置已保存
**When** 再次进入配置页查看
**Then** API Key/Token 字段显示为掩码（如 `****8a3f`），不可直接复制原始值

**Given** 致远 OA 版本为 A8 V5 8.0+（P0）
**When** 管理员在版本选择中选择"A8 V5"
**Then** 系统使用 ZhiyuanA8V5Adapter 发起请求，符合 A8 V5 API 协议规范

**Given** 致远 OA 版本为 A6（P1 适配）
**When** 管理员在版本选择中选择"A6"
**Then** 系统切换至 ZhiyuanA6Adapter，使用对应协议格式发起请求（P1 阶段实现）

**异常情况：**
- 如果 OA 服务地址格式不合法（非 URL 格式），前端实时提示"请输入有效的服务地址（如 http://oa.example.com）"
- 如果保存时任何必填字段为空，系统提示具体缺少哪个字段，阻止保存
- 如果 OA 表单 ID 格式不符合致远规范，系统在测试阶段给出提示

### Story 3.2：OA 接口连通性测试

As a 应用配置管理员，
I want 在保存 OA 配置后立即测试接口连通性，
So that 及时发现配置错误（地址错误、凭证失效），避免在表单发布后才发现回传失败。

**前提条件：**
- OA 集成基本配置已保存（Story 3.1）
- 致远 OA 服务可从 AI-Hub 网络访问

**操作步骤：**
1. 在 OA 集成配置页点击"测试连通性"按钮
2. 等待系统发起 Ping 请求（超时上限 10s）
3. 查看测试结果（成功/失败 + 具体原因）
4. 如连通性通过，可进一步点击"Mock 数据测试"验证写入接口

**成功标志：**
- 连通性测试显示绿色"连接成功"及 OA 版本号
- Mock 数据测试显示"写入测试成功，OA 已收到测试记录"
- 配置错误时显示具体错误原因，引导修正

**页面描述：**
- **测试区域**：在 OA 集成配置页底部，包含"测试连通性"按钮和"Mock 数据测试"按钮；测试结果区域显示状态图标（绿色✓成功/红色✗失败）+ 结果文字说明 + OA 版本信息（成功时）或错误详情（失败时）

**Acceptance Criteria:**

**Given** 应用配置管理员填写完 OA 配置
**When** 点击"测试连通性"
**Then** 系统向 OA 接口发起 Ping 请求（超时 10s），测试通过时显示绿色"连接成功"及 OA 版本信息；失败时显示红色错误原因（连接超时 / 401 鉴权失败 / 404 接口不存在）

**Given** OA 接口连通性测试通过
**When** 管理员点击"Mock 数据测试"
**Then** 系统发送一条预设的测试数据至 OA 写入接口，OA 返回成功响应，页面显示"写入测试成功，OA 已收到测试记录"

**Given** Mock 数据测试执行
**When** OA 写入接口返回错误（如字段格式不匹配）
**Then** 页面展示 OA 返回的具体错误码和错误描述，引导管理员核查字段映射配置

**异常情况：**
- 如果 Ping 请求超过 10s 无响应，提示"连接超时（>10s），请检查 OA 服务地址或网络连通性"
- 如果 OA 返回 401，提示"鉴权失败，请检查 API Key/Token 是否有效或已过期"
- 如果 Mock 数据测试通过但连通性测试失败，提示矛盾情况（不应发生），建议重新保存配置后测试

### Story 3.3：供应商数据异步回传 OA 队列

As a 系统，
I want 将供应商提交的表单数据通过 RabbitMQ 异步队列回传至 OA，并在失败时自动重试，
So that 数据回传不阻塞供应商提交操作，且在 OA 短暂故障时不丢失数据。

**前提条件：**
- OA 集成配置完成且连通性测试通过（Story 3.1 + 3.2）
- RabbitMQ 已配置 `aihub.oa.write` Exchange/Queue 及对应 DLQ

**操作步骤（系统自动执行，无需用户手动操作）：**
1. 供应商提交表单 → 数据写入 AI-Hub 数据库
2. 系统将提交数据发布至 `aihub.oa.write` 队列（含幂等 `messageId`）
3. 供应商端立即收到"提交成功"确认页面
4. 消费者异步从队列取出消息，调用 OA 写入接口
5. 成功：ACK 消息，记录成功日志
6. 失败：按指数退避重试（1min → 5min → 15min），3 次失败后转入 DLQ

**成功标志：**
- 供应商提交后立即收到成功确认，不等待 OA 回传
- OA 在 ≤10 分钟内收到供应商提交数据
- 失败记录进入 DLQ 后，数据管理员可在 Epic 7 的监控面板中查看并手动重试

**业务规则：**
- 回传消息包含 `messageId`（UUID），消费者处理前先校验 Redis 幂等 key，重复消息直接 ACK
- OA 写入成功后在 `interface_call_log` 记录：消息 ID、供应商标识、OA 表单 ID、响应时间、状态=成功
- 进入 DLQ 的消息保留 7 天（可配置），等待人工处理

**Acceptance Criteria:**

**Given** 供应商提交表单后
**When** 系统将提交数据放入 RabbitMQ `aihub.oa.write` 队列
**Then** 供应商端立即收到"提交成功"确认，不等待 OA 回传结果；队列消息包含 `messageId`（UUID）用于幂等校验

**Given** OA 回传消费者从队列取出消息
**When** 调用致远 OA 写入接口成功
**Then** 消息 ACK，`interface_call_log` 表记录状态=成功、响应时间、OA 返回码；整体回传延迟 ≤10 分钟

**Given** OA 回传接口调用失败（超时/5xx）
**When** 消费者捕获异常
**Then** 消息按指数退避重试（1min → 5min → 15min），最多 3 次；3 次均失败后消息转入 `aihub.oa.write.dlq` 死信队列，`interface_call_log` 记录状态=失败、错误码、错误原因

**Given** 同一条提交数据因网络重传被重复入队
**When** 消费者处理该消息
**Then** 通过 `messageId` Redis 幂等校验，检测到重复消息直接 ACK，不产生重复 OA 写入记录

**异常情况：**
- 如果 OA 返回 4xx（客户端错误，如字段格式不符），不进行重试（非临时故障），直接记录失败并转入 DLQ，同时在日志中标注"字段格式错误，需人工介入"
- 如果 RabbitMQ 服务临时不可用，供应商提交数据先写入数据库，系统每 5 分钟尝试恢复入队，最长等待 30 分钟

### Story 3.4：短信通知服务接口配置

As a 应用配置管理员，
I want 在应用内配置短信通知服务的接口参数及消息模板，
So that 后续通知规则触发时，系统能通过指定的短信服务商发送通知。

**前提条件：**
- 应用配置管理员已进入应用工作台（Story 2.3）
- 已获取短信服务商的 AccessKey、SecretKey、签名名称和模板 CODE

**操作步骤：**
1. 进入应用工作台的"集成配置 > 短信配置"模块
2. 选择短信服务商（阿里云 / 腾讯云 / 自定义）
3. 填写 AccessKey、SecretKey、短信签名名称、模板 CODE
4. 填写测试手机号，点击"发送测试短信"
5. 确认手机收到测试短信后，点击"保存配置"

**成功标志：**
- 配置保存成功，AK/SK 加密存储显示为掩码
- 测试手机号收到测试短信，页面显示"发送成功"及 MessageId
- 后续切换服务商时，无需重启服务即可生效

**页面描述：**
- **短信配置页**：服务商选择（下拉：阿里云/腾讯云/自定义）→ 动态展示对应配置字段；测试区域含手机号输入框和"发送测试短信"按钮；测试结果区域显示成功/失败及 MessageId 或错误原因

**Acceptance Criteria:**

**Given** 应用配置管理员进入短信配置页
**When** 选择短信服务商（阿里云 / 腾讯云 / 自定义），填写 AccessKey、SecretKey、签名名称、模板 CODE
**Then** 配置保存成功，AK/SK 加密存储，页面展示为掩码

**Given** 短信配置已填写
**When** 点击"发送测试短信"，输入测试手机号
**Then** 目标手机号收到测试短信，页面显示"发送成功"及短信服务商返回的 MessageId；失败时显示具体错误（余额不足/签名未审核/手机号格式错误）

**Given** 系统已支持阿里云短信
**When** 管理员切换至腾讯云短信并填写对应参数
**Then** 系统通过 `SmsClientFactory` 动态切换实现，无需重启服务，发送测试通过

**异常情况：**
- 如果短信服务商余额不足，测试短信发送失败，提示"发送失败：账户余额不足，请充值后重试"
- 如果签名未通过审核，提示"发送失败：短信签名未审核通过，请在服务商控制台检查签名状态"
- 如果模板 CODE 不存在，提示"发送失败：模板不存在，请检查模板 CODE 是否正确"

### Story 3.5：邮件通知服务接口配置（P1）

As a 应用配置管理员，
I want 配置邮件通知服务（SMTP 或邮件服务商 API）及邮件模板，
So that 支持通过邮件渠道向供应商和内部人员发送通知。

**前提条件：**
- 应用配置管理员已进入应用工作台（Story 2.3）
- 已获取 SMTP 服务器配置或邮件服务商 API Key（如 SendGrid、阿里云邮件推送）
- 此为 P1 功能，需系统已完成短信配置（Story 3.4）作为参考基础

**操作步骤：**
1. 进入应用工作台的"集成配置 > 邮件配置"模块（P1 功能）
2. 选择邮件发送方式（SMTP / 服务商 API）
3. 填写对应配置（SMTP：服务器地址/端口/账号/密码；API：API Key/发件人地址）
4. 配置邮件模板（主题 + 正文，支持动态变量）
5. 填写测试邮箱，点击"发送测试邮件"
6. 确认邮件收到后，点击"保存配置"

**成功标志：**
- 配置保存成功，发件密码/API Key 加密存储显示为掩码
- 测试邮箱收到测试邮件，格式正确，动态变量替换为示例值
- 后续通知规则选择邮件渠道时，可正常发送

**Acceptance Criteria:**

**Given** 应用配置管理员进入邮件配置页（P1 功能）
**When** 填写 SMTP 服务器地址、端口、发件人账号/密码（或服务商 API Key），配置邮件模板（主题、正文，支持动态变量）
**Then** 配置保存成功，发件人密码/API Key 加密存储

**Given** 邮件配置已完成
**When** 点击"发送测试邮件"，输入测试邮箱
**Then** 目标邮箱收到测试邮件，页面显示"发送成功"；失败时展示 SMTP 错误码或服务商错误描述

**异常情况：**
- 如果 SMTP 服务器连接被拒绝（端口不可达），提示"SMTP 连接失败，请检查服务器地址和端口配置"
- 如果 SMTP 账号/密码错误，提示"SMTP 鉴权失败，请检查发件人账号和密码"
- 如果邮件被目标邮箱服务商拦截为垃圾邮件，测试页面仍显示"发送成功"（已离开 AI-Hub 系统），提醒管理员检查收件箱的垃圾邮件文件夹
